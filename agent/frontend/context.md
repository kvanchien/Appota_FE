# Frontend Agent Context

## Purpose

This file defines the fixed context for the Frontend Agent working inside `Appota_FE`.

The agent must read this file before planning or coding any frontend task.

## Project

Project name: Appota Chatbot QA вЂ” Frontend.

Frontend scope includes:

- Player-facing chat screen.
- Admin conversation logs.
- Admin Knowledge Base management.
- API integration with AppotaBackend.
- SSE streaming rendering for chatbot responses.

This is a small 24-hour delivery project. Prioritize functional correctness, clean code, stable API integration, and passing validation commands over complex UI design.

## Core Product Features

## 1. Player Chat

Player sends a message and receives a streamed chatbot response.

The chatbot answer is generated by the backend using Knowledge Base context and Grok API output.

The client must render the assistant response progressively as stream chunks arrive.

## 2. Conversation Logs

Admin can view all conversation sessions.

Admin can open one conversation and see the full message history.

## 3. Knowledge Base Management

Admin can:

- View Q&A entries.
- Create a new Q&A pair.
- Edit an existing Q&A pair.
- Delete a Q&A pair.

After Knowledge Base changes, the next chat request should use updated backend data.

The frontend should not aggressively cache Knowledge Base data unless explicitly requested.

## Chat Streaming Rule

Chat is not webhook-based.

Chat is not WebSocket-based.

Chat is not polling-based.

Chat uses one `POST /api/chat` request and receives a `text/event-stream` response.

The frontend must read the response body as a stream and render assistant output token-by-token or chunk-by-chunk.

Expected stream event format:

`data: {"token":"Xin"}`

`data: {"token":" chГ o"}`

`data: {"token":"!"}`

`data: {"done":true}`

Frontend behavior:

- Send user message to backend.
- Show the user message immediately.
- Create or show an assistant message placeholder.
- Append each received `token` to the current assistant message.
- Stop loading state when `{ "done": true }` is received.
- Show a readable error state if stream fails.
- Do not wait for the full response before rendering.
- Do not create one assistant bubble per token.

Because the API uses `POST /api/chat` with JSON body, prefer `fetch()` with `response.body.getReader()`.

Do not replace this flow with `EventSource` unless the backend API contract changes to GET-based SSE.

## API Contract Summary

Non-stream JSON responses should use this format:

{
"success": true,
"data": {},
"message": "optional message"
}

Failed responses should use:

{
"success": false,
"data": null,
"message": "error message"
}

Chat stream is the exception because it returns `text/event-stream`.

## Important Endpoints

`POST /api/session`
Create a new chat session.

`POST /api/chat`
Send player message and receive streamed assistant response.

`GET /api/chat/:sessionId`
Get chat history for one session.

`GET /api/conversations`
Get admin conversation list.

`GET /api/conversations/:id`
Get admin conversation detail.

`GET /api/knowledge`
Get all Knowledge Base entries.

`POST /api/knowledge`
Create a Knowledge Base entry.

`PUT /api/knowledge/:id`
Update a Knowledge Base entry.

`DELETE /api/knowledge/:id`
Delete a Knowledge Base entry.

## Frontend Agent Scope

The Frontend Agent may work on:

- React components.
- React hooks.
- API wrapper functions.
- SSE parsing.
- UI states.
- Form validation.
- Admin table behavior.
- Route wiring.
- Styling with Tailwind CSS.
- Ant Design component usage.
- Build and lint fixes.

The Frontend Agent must not work on:

- Backend controller/service/model code.
- MongoDB schema.
- Grok API server integration.
- Express route implementation.
- Server deployment configuration.
- Authentication unless explicitly requested.

## Development Priorities

For each frontend task, prioritize in this order:

1. Correct API behavior.
2. Correct user-visible behavior.
3. Stable state management.
4. Clear loading, empty, and error states.
5. Responsive layout.
6. Clean and simple component structure.
7. Passing lint and build.
8. UI polish.

Do not spend time on advanced animations, complex design systems, or over-engineered abstractions unless explicitly requested.

## Validation Commands

Before finishing a frontend task, run the available validation commands when possible:

`npm run lint`

`npm run build`

If formatting scripts exist, also run:

`npm run format:check`

Do not claim validation passed unless the command was actually run.

If a command fails, record the failure and cause in `agent/frontend/handoff.md`.

## Current Plan Rule

Before coding any task, the agent must update:

`Appota_FE/agent/frontend/current-plan.md`

The current plan must be generated from the task prompt and include:

- Current task.
- Affected feature.
- Scope.
- Files to inspect.
- Files likely to modify.
- Implementation steps.
- Validation commands.
- Out-of-scope items.

Do not code before current plan is updated.

## Handoff Rule

After each task, the agent must update:

`Appota_FE/agent/frontend/handoff.md`

The handoff must include:

- Task summary.
- Files changed.
- Validation commands run.
- Result.
- Known issues.
- Backend/API dependencies.
- Suggested next step.

## Constraints

Keep changes small and reviewable.

Do not mix unrelated changes in one task.

Do not refactor large parts of the frontend unless explicitly requested.

Do not change API contracts from the frontend side.

Do not add new dependencies unless the task explicitly requires it.

Do not implement fake data flow if real API integration is expected.

## Completion Definition

A frontend task is complete only when:

- The requested behavior is implemented.
- Related UI state is handled.
- API errors are handled.
- No unrelated files are changed.
- `current-plan.md` reflects the task.
- `handoff.md` records the result.
- Lint/build status is known.
